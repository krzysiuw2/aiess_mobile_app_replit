# AI Optimization Agent — Safety and Rollback

This document describes the safety constraints, validation pipeline, health checks, and rollback mechanisms that ensure the AI Optimization Agent never compromises grid safety, equipment integrity, or regulatory compliance.

---

## 1. Priority Allocation Model

The agent uses a **priority-based rule hierarchy** (P6–P9) to ensure safety rules always take precedence over AI-generated optimizations. Lower numbers indicate higher priority.

| Priority | Name | Description | AI-Modifiable |
|----------|------|-------------|---------------|
| **P9** | Safety-only | Hard safety rules: SoC limits, power limits, grid capacity, export control | **Never** |
| **P8** | High-priority AI | Bell curve (Rynek Mocy), capacity market participation | Yes |
| **P7** | Mid-priority AI | Arbitrage, PV self-consumption, peak shaving | Yes |
| **P6** | Low-priority AI | Intraday adjustments, minor tweaks | Yes |

### 1.1 Enforcement

- **P9 rules** are defined in site configuration and enforced by the device firmware / IoT Shadow. The agent **never modifies** P9 rules.
- **P8–P6 rules** are generated by the agent and validated against P9 constraints before application.
- On conflict, **P9 always wins** on the device; the agent is designed to never propose rules that violate P9.

---

## 2. Hard Constraints

All AI-generated rules must satisfy the following hard constraints before being applied.

### 2.1 SoC Limits

| Constraint | Source | Description |
|------------|--------|-------------|
| **SoC min** | P9 safety rules | Minimum state of charge (%). Never discharge below this. |
| **SoC max** | P9 safety rules | Maximum state of charge (%). Never charge above this. |

Values are read from `site_config` or device configuration. The agent uses these limits in the optimization engine and constraint validator.

### 2.2 Power Limits

| Constraint | Source | Description |
|------------|--------|-------------|
| **Max charge power (kW)** | Site config | Maximum charging power; rules cannot exceed this. |
| **Max discharge power (kW)** | Site config | Maximum discharging power; rules cannot exceed this. |

### 2.3 Grid Capacity

| Constraint | Source | Description |
|------------|--------|-------------|
| **Grid capacity (kVA)** | `moc_zamowiona` from site config | Maximum import from the grid. Exceeding this can trigger penalties or disconnection. |

### 2.4 Export Control

| Constraint | Source | Description |
|------------|--------|-------------|
| **Export allowed** | Site config | Boolean flag. If `false`, the agent must never propose discharge that results in net export to the grid. |

---

## 3. Safety Margins

To avoid operating at the edge of constraints, the agent applies configurable safety margins.

### 3.1 Export Buffer

- **Value:** 1 kW above 0 (net export)
- **Purpose:** Prevents inadvertent export when load drops slightly. The agent treats "0 kW export" as a hard limit and adds a 1 kW buffer so that minor fluctuations do not cause illegal export.

### 3.2 Import Buffer

- **Value:** 2–3% below `moc_zamowiona` (ordered power / grid capacity)
- **Purpose:** Prevents exceeding grid capacity due to measurement error or transient spikes. The agent targets import at 97–98% of capacity during peak shaving.

---

## 4. Schedule Snapshot

Before every **daily** and **weekly** run, the agent stores a **schedule snapshot** in `aiess_agent_state.schedule_snapshot`.

### 4.1 Snapshot Contents

- Full IoT Shadow schedule state (all active rules)
- Timestamp of snapshot
- Site ID

### 4.2 When Snapshots Are Taken

| Agent | When |
|-------|------|
| Daily Planner | Before applying new daily rules |
| Weekly Strategist | Before any schedule changes from weekly guidance |

### 4.3 Purpose

The snapshot is used for **rollback** when the Intraday Adjuster detects critical health violations. Restoring the snapshot returns the device to the last known good state.

---

## 5. Health Check Conditions (Intraday)

The Intraday Adjuster runs health checks on every invocation. If any of the following conditions are detected, the agent triggers a **rollback**:

| Condition | Description | Severity |
|-----------|-------------|----------|
| **Grid import > capacity** | Real-time import exceeds `moc_zamowiona` | Critical |
| **SoC < min** | Battery state of charge below minimum | Critical |
| **Illegal export** | Net export when `export_allowed` is false | Critical |
| **Bell curve violation** | Discharge during Rynek Mocy restricted hours | Critical |

### 5.1 Health Check Flow

1. Intraday Adjuster fetches current telemetry (load, PV, SoC, grid import/export).
2. Compares against constraints and P9 rules.
3. If any critical violation is detected → **rollback**.
4. If no violation → proceed with normal intraday logic (adjustments or replan).

---

## 6. Rollback Procedure

When a critical health violation is detected:

### 6.1 Steps

1. **Retrieve snapshot** from `aiess_agent_state.schedule_snapshot`.
2. **Restore to IoT Shadow** via the Schedules API (PUT/PATCH to overwrite current schedule).
3. **Log incident** in `aiess_agent_decisions` with:
   - `agent_type: 'intraday'`
   - `status: 'rolled_back'`
   - `reasoning`: Description of the violation
   - `lesson_learned`: Extracted lesson for future runs
4. **Increment** `performance_30d.rollbacks_count` in `aiess_agent_state`.
5. **Optionally** create a `rollback_alert` notification for the user.

### 6.2 Schedules API Integration

- The agent uses the Schedules API to read and write schedule rules.
- Rollback = full overwrite of the device schedule with the snapshot content.
- The API forwards the update to the IoT Shadow.

---

## 7. Decision Validation

Every AI-generated rule passes through a **constraint validator** before application.

### 7.1 Validation Pipeline

1. **Optimization Engine** produces structured outputs (charge/discharge windows, power levels).
2. **Constraint Validator** checks:
   - SoC min/max respected over the horizon
   - Max charge/discharge power not exceeded
   - Grid capacity not exceeded
   - Export buffer and import margin applied
   - Export allowed flag respected
3. If validation fails → rules are **rejected** or **adjusted** to satisfy constraints.
4. Only validated rules are sent to the Schedules API.

### 7.2 Validator Behavior

- **Reject:** Rules that cannot be fixed (e.g. impossible SoC trajectory) are rejected; the agent logs the failure and may retry with different parameters.
- **Adjust:** Rules that slightly exceed limits (e.g. 105% of max power) are clamped to the limit.

---

## 8. Emergency Override

### 8.1 Priority Hierarchy on Device

On the BESS controller / IoT device:

- **P9 safety rules** are always enforced first.
- P8–P6 rules are applied only if they do not conflict with P9.
- The device firmware treats P9 as immutable by the agent.

### 8.2 Agent Awareness

- The agent is aware of P9 rules and never proposes rules that would conflict.
- In semi-automatic mode, users can reject or modify AI proposals before application.
- In automatic mode, the constraint validator is the last line of defense before rules reach the device.
