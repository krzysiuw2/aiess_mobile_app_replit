## CONTEXT: What You Are Building

You are implementing a **new mathematical optimization engine** for the AIESS Battery Energy Storage System (BESS) platform. This replaces the current architecture where an LLM (Bedrock Claude) generates schedule rule JSON directly. The new architecture:

1. A **Python Lambda** runs deterministic math to produce 3 complete strategy packages (aggressive/balanced/conservative), each containing validated schedule rules
2. The **LLM (Bedrock)** is a **mandatory validation and selection layer** that picks strategy A/B/C and optionally adjusts 1-2 parameters. It no longer generates rule JSON.
3. Each strategy package includes a **forecast energy flow** (hour-by-hour SoC, grid, battery, PV projections) for client-facing charts
4. The **Schedules API** validates rules against schema before writing to IoT Shadow
5. The **edge device** receives rules via IoT Shadow and executes them at 1 Hz

### What Changed (Big Picture)

| Before (v1.x) | After (v2.0) |
|---|---|
| Optimization engine produces "hints" (windows, recommendations) | Engine produces **complete, validated rule sets** in compact format |
| LLM generates full rule JSON (error-prone, expensive) | LLM only **validates and selects** strategy A/B/C (~50 tokens output) |
| No schema validation before deployment | **Mandatory schema validation** in Schedules API |
| Format mismatches between AI output and edge device | **Single canonical compact format** everywhere |
| ~$0.30/day/site LLM cost | ~$0.02/day/site LLM cost |

   - Rejects rules that fail validation with clear error messages

2. **Add schema validation gate**
   - Every rule must pass validation before being written to IoT Shadow
   - Validate: required fields (`id`, `a.t`), valid action types, SoC ranges (0-100), time ranges (0-2359), grid operator values (`gt`/`lt`/`eq`/`gte`/`lte`)

### Phase 2: New Optimization Engine (Core Work)

**Goal**: Build the Python Lambda that implements the 9-step mathematical pipeline from `03_OPTIMIZATION_ENGINE.md`.

1. **Create new Lambda**: `aiess-optimization-engine-v2` (Python 3.11+, containerized)
   - Dependencies: `numpy` (for statistics), `influxdb-client` (for InfluxDB queries), `boto3` (for DynamoDB)
   - Environment: same InfluxDB and DynamoDB access as existing engine

2. **Implement the 9-step pipeline** (follow `03_OPTIMIZATION_ENGINE.md` precisely):
   - Step 1: Compute Hourly Net Grid
   - Step 2: Price Window Optimizer
   - Step 3: PV Surplus Calculator (with surplus shape output)
   - Step 4: Peak Shaving Calculator with **Confidence Bands** (query historical peaks from InfluxDB, compute empirical percentiles, configurable `peak_confidence` default 0.99)
   - Step 5: Trade-off Resolver (roomForPV + peak shaving readiness + holiday awareness)
   - Step 6: Strategy Generator (3 strategies: aggressive/balanced/conservative)
   - Step 7: SoC Trajectory Simulator (validate no SoC/grid violations)
   - Step 8: Rule Builder (output canonical compact format)
   - Step 9: Forecast Energy Flow (hour-by-hour projections for charts)

3. **Implement effective limit calculation** (CRITICAL - see `03_OPTIMIZATION_ENGINE.md` Section 2.1):
   - 3 independent limit types: `moc_zamowiona_kw` (import only), `grid_capacity_kva` (both), `export_limit_kw`/`import_limit_kw` (P9 user-configured)
   - `effective_import_limit = min(moc_zamowiona, grid_capacity, import_limit)` -- skip missing values
   - `effective_export_limit = min(grid_capacity, export_limit)` -- NO moc_zamowiona for export
   - All rule generation uses effective limits, not raw values

4. **Output structure per strategy**:
   ```json
   {
     "name": "balanced",
     "rules": [ /* compact S3/S6 format rules */ ],
     "forecast": {
       "horizon_hours": 24,
       "hourly": [ /* per-hour: soc, battery_kw, grid_kw, pv_kw, load_kw, costs */ ],

     "risk": "low"

   ```

5. **Rules that MUST always be present** (regardless of forecast):
   - P6: PV surplus capture (grid power < 0 triggers charge)
   - P8: Peak shaving (grid power > threshold triggers discharge)
   - See `04_REAL_WORLD_PATTERNS.md` Section 4 for exact rule structures

## THINGS TO BE AWARE OF

### Edge Device Expectations

The edge device (Robustel EG5120, C code) expects rules in the **compact S3/S6 format** as defined in `02_CANONICAL_RULE_SCHEMA.md`. Key points:

- Action types: `ch`, `dis`, `sb`, `sl`, `ct`, `dt` (abbreviated)
- Time: HHMM integers (e.g., 1430 = 14:30)
- Conditions: flattened (`ts`, `te`, `sm`, `sx`, `gpo`, `gpv`)
- Weekdays: array of ints [1-7] where 1=Mon, or strings like `"wkd"`, `"wke"`
- Goal-based actions (`ct`/`dt`) require: `soc` (target), `str` (strategy: `eq`/`agg`/`con`), `maxp` (max power)
- Grid constraints in actions: `maxg` (max grid import while executing), `ming` (min grid, i.e., max export)
- `valid_until` (`vu`): MANDATORY for all cloud-generated rules (Unix timestamp)
- Source tag: `"s": "ai"` for engine-generated, `"s": "man"` for manual/app-created

### P9 Site Limits Are NOT Generated By The Engine

P9 site limits (`hth`/`lth`) are set by the user/installer via the mobile app. The optimization engine reads them as INPUT constraints but never generates or modifies them. The edge device's P9 active controller enforces them as last-resort protection.

### Polish Holidays

The forecasting system already includes Polish holidays in load predictions. The optimization engine's trade-off resolver also needs holiday awareness for battery planning (treats holidays as weekend-like days). See `04_REAL_WORLD_PATTERNS.md` Section 6 for the complete list of 13 Polish public holidays and multi-day patterns.

### The Forecast Engine Already Produces What We Need

The existing `aiess-forecast-engine` Lambda already writes `pv_forecast_kw` and `load_forecast_kw` to InfluxDB (`energy_simulation` measurement in `aiess_v1_1h` bucket). The new optimization engine reads this data -- it does NOT need to generate its own forecasts.

### Existing Agent Decisions Table

The `aiess_agent_decisions` table already stores decision logs with `predicted_outcome` and `actual_outcome` fields. Extend this to also store the selected strategy's `forecast` object for chart rendering in the app.

### IoT Shadow Write Path

Rules flow: Schedules API Lambda -> IoT Shadow (`desired` state) -> Edge device `shadow_sync` process picks up delta -> merges into `/var/lib/aiess/schedules.json` -> Schedule engine hot-reloads.

The shadow document structure follows the compact format:
```json
{
  "state": {
    "desired": {
      "v": "1.2",
      "mode": "automatic",
      "sch": {
        "p_6": [ /* P6 rules */ ],
        "p_7": [ /* P7 rules */ ],
        "p_8": [ /* P8 rules */ ],
        "p_9": [ /* P9 rules (user-configured, untouched by engine) */ ]
      }
    }
  }
}
```

### Backward Compatibility

- Existing manual rules (source `"man"`) must continue to work unchanged
- The engine only writes P6-P8 rules. P9 (user), P5 (fallback), P10/P11 are not touched.
- Old rules without a `vu` (valid_until) field are treated as permanent (local/manual rules)
- The edge device already ignores unknown fields, so adding new fields is safe

### Testing Strategy

1. **Unit test the math pipeline** with known inputs and expected outputs (the numerical example in `03_OPTIMIZATION_ENGINE.md` Section 5 is a good starting test case)
2. **Test with real site data** from `domagala_1` -- fetch actual TGE prices, forecasts, and site_config
3. **Test the confidence band calculator** with at least 30 days of historical peak demand data
4. **Test the LLM selector** with mocked strategy packages -- verify it returns valid JSON and falls back to B on failures
5. **End-to-end**: Deploy to test site, observe one full daily cycle, compare forecast vs actual

## IMPORTANT DESIGN DECISIONS (Do Not Override)

These decisions were made after extensive discussion with the system architect (the edge device developer). Do not change them without explicit approval:

1. **The engine generates P6, P7, P8 rules ONLY**. P9 is user-configured. P5 is fallback. P10/P11 are system-controlled.
2. **P9 site limits are INPUT constraints**, not outputs. Never generate or modify P9 rules.
3. **The LLM is MANDATORY** as a validation layer. It always runs. Strategy B is the fallback ONLY on LLM failure.
4. **Rules use compact S3/S6 format** everywhere. No verbose format, no translation layers.
5. **`valid_until` is MANDATORY** for all cloud-generated rules. This is the key to offline robustness.
6. **Peak shaving rules are ALWAYS generated**, even when forecasts show no risk. The cost of a moc_zamowiona breach in Poland far exceeds the cost of a defensive rule.
7. **PV surplus capture rules are ALWAYS generated**, even when forecasts predict zero surplus.
8. **Confidence bands** (not fixed safety margins) determine the peak shaving reserve SoC. Default confidence: 99%.
9. **3 limit types** must be distinguished: moc_zamowiona (import only), grid_capacity (both), P9 export/import limits (direction-specific). Always take `min()` of applicable constraints.
10. **Forecast energy flow** is included in every strategy package. This is not optional -- it enables client-facing charts.
